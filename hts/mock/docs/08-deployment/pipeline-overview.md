# CI/CD パイプライン概要

## 概要

本プロジェクトでは、**GitHub Actions** を使用した CI/CD パイプラインを構築します。
`main` ブランチへのマージで本番環境へ、`develop` ブランチへのマージで開発環境へ自動デプロイを実行します。

## CI/CD の目的

1. **品質保証**: 自動テストによるバグの早期発見
2. **効率化**: 手動デプロイ作業の削減
3. **一貫性**: デプロイプロセスの標準化
4. **迅速性**: コードから本番環境への高速なリリース
5. **安全性**: テスト合格後のみデプロイ

## パイプライン構成

### 全体フロー

```
コード変更
  ↓
Git Push
  ↓
GitHub Actions トリガー
  ↓
┌─────────────────────────────────┐
│ CI: 継続的インテグレーション      │
│ - Lint チェック                  │
│ - 単体テスト                     │
│ - ビルド                         │
│ - セキュリティスキャン            │
└─────────────────────────────────┘
  ↓ (テスト成功)
┌─────────────────────────────────┐
│ CD: 継続的デプロイ                │
│ - 本番/開発環境へデプロイ         │
│ - 動作確認                       │
│ - 通知                           │
└─────────────────────────────────┘
```

## パイプライン詳細

### 1. Pull Request 作成時

**トリガー**: PR が作成・更新されたとき

**目的**: マージ前の品質チェック

**ステップ**
```yaml
1. Checkout コード
2. キャッシュ復元（依存関係）
3. 依存関係インストール
4. Lint チェック
   - ESLint（フロントエンド）
   - Prettier（フロントエンド）
   - PHPStan（バックエンド）
   - Laravel Pint（バックエンド）
5. 単体テスト実行
   - Vitest（フロントエンド）
   - PHPUnit（バックエンド）
6. ビルドテスト
   - Next.js ビルド
   - Laravel アセットビルド
7. テストカバレッジレポート生成
8. PR にコメント（結果通知）
```

### 2. develop ブランチへのマージ時

**トリガー**: develop ブランチへの push

**目的**: 開発環境への自動デプロイ

**ステップ**
```yaml
1. Checkout コード
2. 依存関係インストール
3. Lint チェック
4. テスト実行（全テスト）
5. ビルド（開発用）
6. 統合テスト実行
7. Docker イメージビルド（オプション）
8. 開発環境へデプロイ
   - フロントエンド（Next.js）
   - バックエンド（Laravel）
   - データベースマイグレーション
9. ヘルスチェック
10. Slack 通知
```

### 3. main ブランチへのマージ時

**トリガー**: main ブランチへの push

**目的**: 本番環境への自動デプロイ

**ステップ**
```yaml
1. Checkout コード
2. 依存関係インストール
3. Lint チェック（厳格モード）
4. テスト実行（全テスト + E2E テスト）
5. セキュリティスキャン
   - npm audit
   - composer audit
   - Snyk（オプション）
6. ビルド（本番用）
   - 環境変数: NODE_ENV=production
   - 最適化ビルド
7. Docker イメージビルド
8. 本番環境へデプロイ
   - バックアップ作成
   - データベースマイグレーション
   - フロントエンドデプロイ
   - バックエンドデプロイ
9. ヘルスチェック
10. Rollback 準備（失敗時）
11. リリースタグ作成
12. Slack 通知（成功/失敗）
13. Sentry へリリース通知
```

## ベストプラクティス

### DO（推奨）

- テストが通過してからデプロイ
- デプロイ前にバックアップを作成
- ヘルスチェックを実装
- デプロイ通知を設定
- ロールバック手順を準備
- Secrets を適切に管理
- キャッシュを活用して高速化

### DON'T（非推奨）

- テストをスキップしてデプロイ
- 本番環境で直接コード編集
- Secrets をコードに含める
- エラー時の通知を無視
- バックアップなしでマイグレーション実行
