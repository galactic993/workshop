# 第5回: AI駆動開発ワークフロー実践

## 学習目標
- Claude Codeを使ったコード生成の手法を理解する
- 設計書からLaravelマイグレーション/モデルを自動生成できる
- プロンプトエンジニアリングの基礎を習得する
- AI駆動開発の可能性と限界を理解する

## 1. AI駆動開発ワークフロー概要

### 従来の開発フロー
```
設計書(Excel) → 人間が読解 → 手動コーディング → レビュー → 修正
```

### AI駆動開発フロー
```
設計書(Excel) → パーサーで変換 → AI入力 → コード自動生成 → 人間レビュー → 微調整
```

### メリット
- 設計書の情報を正確にコードに反映
- 繰り返し作業(マイグレーション、モデル定義)の自動化
- 一貫性のあるコード生成

### 注意点
- AI生成コードは必ず人間がレビューする
- ビジネスロジックの細部はAIに任せすぎない
- 生成コードのテストは必須

**効果的なプロンプト例:**
```
以下のテーブル定義書(Markdown形式)を元に、Laravel 11のマイグレーションファイルを生成してください。

要件:
- テーブル名はスネークケースを使用
- 各カラムのデータ型をLaravel Schema Builderのメソッドに対応付け

データ型の対応:
- VARCHAR → string
- CHAR → char
- INTEGER → integer
- SMALLINT → smallInteger
- BIGINT → bigInteger
- DECIMAL → decimal
- TEXT → text
- DATE → date
- TIMESTAMP → timestamp
- BOOLEAN → boolean

[ここにMarkdownのテーブル定義を貼り付け]
```

## 5. プロンプトエンジニアリングの基礎

### 効果的なプロンプトの構造
```
1. 役割の指定: 「あなたはLaravel 11の専門家です」
2. コンテキスト: 設計書のMarkdownデータ
3. 具体的な指示: 何を生成するか
4. 制約条件: 命名規則、コーディング規約
5. 出力形式: ファイルパス、コードフォーマット
```

### プロンプトのコツ

#### Do(推奨)
- 具体的なデータ型の対応表を提示する
- 出力形式を明確に指定する
- 1回の依頼で1種類の成果物を生成する
- 制約条件を箇条書きで列挙する

#### Don't(非推奨)
- 「いい感じに作って」のような曖昧な指示
- 一度に大量のファイルを生成させる
- AIの出力をレビューせずに使う
- エッジケースの処理をAI任せにする

### テンプレート化のススメ
```markdown
## マイグレーション生成プロンプトテンプレート

以下のテーブル定義を元に、Laravel 11のマイグレーションを生成してください。

### テーブル定義
{テーブル定義のMarkdown}

### 要件
- Laravelの命名規則に従う
- タイムスタンプカラム(created_at/updated_at)を含む
- 外部キー制約を正確に設定
- インデックスが必要なカラムにはindex()を追加

### 出力形式
- ファイル名: `YYYY_MM_DD_HHMMSS_create_{テーブル名}_table.php`
- Laravel 11の無名クラス形式
```

## 7. AI駆動開発の可能性と限界

### 可能性
- **定型的なコード生成**: マイグレーション、モデル、CRUD Controller
- **設計書の整合性チェック**: 人間より高速・正確
- **ドキュメント生成**: API仕様書、テスト仕様書
- **テストコード生成**: 基本的なユニットテスト

### 限界
- **ビジネスロジック**: 業務知識が必要な処理はAIだけでは不十分
- **パフォーマンス最適化**: 実際の負荷パターンに基づく最適化
- **セキュリティ**: 認証・認可の細かい設計
- **UI/UX**: デザインの感性的な部分

### ベストプラクティス
1. AIに生成させる → 人間がレビュー → 微調整
2. 設計書の品質がAI出力の品質を決める
3. 小さい単位で生成し、確認しながら進める
4. 生成コードのテストは必ず書く

## まとめ
- 設計書をMarkdownに変換することでAIの入力にできる
- プロンプト設計が生成品質を大きく左右する
- AI生成コードは必ず人間がレビューする
- 定型的な作業から始めて、徐々にAI活用範囲を広げる
