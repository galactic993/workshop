# セイコーソリューションズ様 AI駆動開発研修プラン（MEZ-154）

## 0. 位置づけ
- 既存の [第1回_AI駆動開発研修](./第1回_AI駆動開発研修.md) をベースに、セイコーソリューションズ様向けの実務適用版へ再構成する。
- 3/12 開始を起点に、全 2〜4 日の範囲で実施できるようにする。
- まずは半田様向けプレ実施を想定し、営業社員・エンジニアへの少人数展開（数名〜10名）を可能にする。

## 1. 研修目的
### 1.1 主要目的
- 4名開発体制での AI 活用を仕組み化し、個人差を減らして一定品質を担保する。
- 要件定義から設計書を起こす流れを定義し、実装・テスト・レビューを高速化する。
- AI 一次レビュー + 人間最終レビューの体制を作り、リードエンジニアのレビュー負荷を削減する。

### 1.2 副次目的
- 社内プレ実施の結果をテンプレ化し、顧客向け「AI駆動開発研修パッケージ」の外販案へ接続する。

## 2. 対象者・前提
- 主対象: AI 未経験を含む開発/営業メンバー（少人数バッチ運用）
- 開発技術: BE=PHP/Laravel、FE=TypeScript（Next.js 想定）
- 設計資産: 要件定義書・設計書（画面/機能単位での記述を想定）
- 実務前提: 要件定義から設計書を作成し、その内容を起点に実装する
- テスト成果物: 顧客提出用の仕様書・実行証跡が必要
- 端末環境: Windows 中心（Mac 混在可）

## 3. スコープ
### 3.1 今回の対象
- コーディングエージェントを使った実務フローの確立
- 要件定義・設計書レビュー自動化（記述漏れ・不整合・整合性逸脱の検知）
- 要件定義/設計書からの実装支援（要件抽出 → タスク化 → コード生成）
- AI 支援前提のテスト実行と、顧客提出用エビデンス整備
- PR レビューの効率化（差分要約・論点整理・一次レビュー導入）

### 3.2 今回の非対象
- 「AI とは何か」の長時間講義（導入は最小限）
- CI/CD・タスク管理ツールの全面刷新（必要時は提案のみ）
- 厳密 TDD への全面移行

## 4. 提供方式（伴走型）
- 週次 MTG + 非同期相談（Slack 等）
- MTG 前: 事前に詰まりポイントを収集
- MTG 中: 高優先トピックの解決、運用フロー調整、設定共同確認
- MTG 後: 必要に応じて短い手順書/テンプレ/動画で補完

## 5. 実施日数オプション（2〜4日）

| 案 | 日数 | 構成 | 想定用途 |
|---|---:|---|---|
| A | 2日 | Day1: 1+2を実施 / Day2: 3実践編を集中実施 | 最短導入、プレ実施向け |
| B | 3日 | Day1: 1+2 / Day2: 実践編(要件→設計) / Day3: 実践編(実装→テスト→レビュー) | 推奨標準 |
| C | 4日 | Day1: 1 / Day2: 2 / Day3-4: 3実践編を段階的に実施 | 未経験者比率が高い場合 |


## 6. 各モジュールの到達目標
### 6.1 モジュール1: AI/LLM 基礎（1〜2時間）
- 生成AI/LLMの基本、機械学習からLLMまでの流れを理解する。
- プロンプト差による出力品質差を体験する。
- ChatGPT/Claude/Gemini の用途差を説明できる状態にする。

### 6.2 モジュール2: 開発研修（初級、2〜3時間）
- Gemini CLI 等での簡易ハンズオンを実施（必要に応じて省略可）。
- 「指示の分解」「制約明示」「レビュー観点指定」の型を身につける。

### 6.3 モジュール3: 開発研修（実践編、2時間 × 複数回）
以下を 1 サイクルとして実務と同じ順序で回す。

1. 要件定義から設計書を作成/補正
2. 設計書レビュー（要件整合性・記述漏れ・仕様矛盾のチェック）
3. 実装タスク化（BE/FE/テストで分割）
4. コード生成と差分レビュー
5. テスト仕様書・テスト結果エビデンス作成
6. PR で AI 一次レビュー → 人間最終判断

## 7. 実践編の標準ワークフロー（チーム運用の型）
### Step 1: 要件定義・設計書レビュー自動化
- A系/C系プロンプトで要件定義・設計書の単体整合性チェック
- B系で画面設計書 × テーブル定義書のクロスチェック
- 出力をレビュー表（error/warning/info）で管理

### Step 2: 実装準備
- 要件定義・設計書から「要件サマリ」「API仕様」「バリデーション一覧」を抽出
- 実装タスクを FE/BE/Test に分解

### Step 3: 実装・レビュー
- Codex を中心に実装、必要時 Claude を補助利用
- PR 単位で AI 一次レビュー（差分要約、論点抽出）
- 人間は設計整合性・業務妥当性に集中して最終判断

### Step 4: テスト証跡整備
- 実態は AI 支援で実行しつつ、顧客提出用に以下を整備
  - テスト仕様書（観点・前提・期待値）
  - テスト結果（実行日時・環境・結果・差分）
  - 不具合票/修正履歴との紐付け

## 8. 主要成果物（Deliverables）
### A. チーム運用の型
- 推奨ツール構成（Codex中心）
- 標準開発手順（要件定義 → 設計書作成 → 実装 → レビュー → テスト → 証跡）
- 役割別テンプレ（要件整理、設計書レビュー、実装、テスト証跡）
- 品質担保チェックリスト（禁止事項含む）

### B. 要件定義・設計書関連
- 1機能分の要件定義 → 設計書サンプル
- 不整合検知ルール（項目番号、参照先整合性、仕様矛盾）

### C. レビュー/テスト効率化
- AI 一次レビュー導入ガイド
- 顧客提出向けテスト仕様書/結果エビデンス作成手順

## 10. セキュリティ・ガバナンス合意項目
研修開始前に以下を合意事項として確定する。

- 利用可能モデル範囲（ローカルのみ / クラウド可）
- 月額利用上限（例: Codex $200 + Claude $100 起点）
- 研修で受け取るデータ範囲（最小1件の要件定義書・設計書、秘匿情報マスキング方針）
- モック共有方法（URL不可時のローカル起動手順、DB含む共有手順）

## 11. 成功指標（KPI）
- 4名体制で共通手順に沿って AI 開発を実施できる
- 要件定義・設計書レビューの手戻り（記述漏れ/不整合）が減少する
- PRレビュー時間が短縮され、AI一次レビューが運用に定着する
- テスト仕様書・結果エビデンスが再現可能な形式で作成される
