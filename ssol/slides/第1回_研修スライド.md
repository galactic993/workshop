# セッション1
## AI・LLMの基礎講座（30分）

---

## AI = LLMの時代

現在の「AI」と呼ばれるものの多くは **LLM（大規模言語モデル）** です

- 大量のテキストデータから言語パターンを学習
- 文章生成、翻訳、要約、**コード生成**が可能
- 人間の指示（プロンプト）に基づいて動作

<div class="accent">

AIは「何でもできる魔法」ではなく、**指示の質で出力が決まる**ツール

</div>

---

## LLMの進化の歴史

| 年 | 出来事 | インパクト |
|----|--------|-----------|
| 2020 | GPT-3 登場 | 初めて「使える」レベルのテキスト生成 |
| 2022 | ChatGPT 公開 | AI革命の引き金。一般ユーザーに普及 |
| 2023 | GPT-4 / Claude 2 | コーディング能力が実用レベルに |
| 2024 | Claude 3.5 / GPT-4o | マルチモーダル（画像+テキスト）対応 |
| 2025 | Claude 4 / コーディングエージェント | **AIが自律的に開発作業を実行** |

> たった5年で「テキスト生成」から「自律開発」へ進化

---

## LLMの仕組み（簡略版）

```
入力（プロンプト）
    ↓
LLM（パターン認識 + 文脈理解）
    ↓
出力（テキスト / コード / 分析結果）
```

**ポイント:**
- 「次に来る可能性が最も高い単語」を予測し続ける
- 学習データに基づいて応答するため、最新情報は苦手
- 同じ質問でも回答が毎回異なることがある（確率的な生成）

---

## LLMが「できること」と「できないこと」

| できること | できないこと |
|-----------|-------------|
| テキスト生成・要約・翻訳 | リアルタイムの最新情報取得 |
| コード生成・デバッグ | 100%正しい回答の保証 |
| データ分析・レポート作成 | 物理的な作業の実行 |
| アイデア出し・壁打ち | 感情や意志を持った判断 |
| 設計書の作成・レビュー | **責任を取ること** |

<div class="warning">

**重要**: AIの出力は必ず人間が検証する。最終判断と責任は常に人間の仕事

</div>

---

<!-- _class: xsmall -->

## 主要なLLM

| LLM | 開発元 | 特徴 | 得意分野 |
|-----|--------|------|----------|
| **OpenAI** | OpenAI | 最も普及・マルチモーダル | 汎用タスク・画像理解・ブラウザ操作型リサーチ |
| **Claude** | Anthropic | 安全性重視・長文理解 | コーディング・分析・長文処理 |
| **Gemini** | Google | 画像生成はできるがコーディングは弱め | 今は無理に使わなくてよい |
| **Grok** | xAI | リアルタイム性・X連携・実ブラウザ寄りの情報収集が強い | AI話題・最新動向の対話型リサーチ |

> まず使うなら OpenAI / Claude / Grok。Geminiは画像生成用途が明確でない限り優先度は低い

---

<!-- _class: small -->

## リサーチ用途で見ると差が大きい

- OpenAIのThinkingモード / Proモード、およびGrokは、実際にブラウザを操作しながら情報収集できる
- 通常のスクレイピングでは対策済みサイトから情報を取りにくいことが多い
- これらはブラウザ経由でページ遷移や画面操作を伴うため、他のAIでは拾いにくい情報まで取れることがある
- とくにAI関連やX起点の最新情報は、Grokの方がリアリティの高い情報に当たりやすい

> **実務上のおすすめ:** 最新動向やAI関連の一次情報を追うなら、まずGrokを使ってみる価値が高い

---

## AIの世代分類

| 世代 | 例 | できること |
|------|----|-----------|
| コード補完 | Copilot初期 | カーソル位置の次の行を予測 |
| チャット | ChatGPT | 質問に回答。**実行はしない** |
| **エージェント** | **Claude Code等** | **複数ファイルを横断して自律的に実装・テスト** |

<div class="accent">

我々が使うのは最新の **「エージェント」世代** — AIが自分で考えて行動する

</div>

---

<!-- _class: small -->

## 主要なコーディングエージェント

| ツール | 開発元 | 形態 | 特徴 |
|--------|--------|------|------|
| **Claude Code** | Anthropic | CLI / IDE / Web | 機能投入の先行例が多い。サブエージェント・MCP・Skills・Hooksなど、業界の基準になりやすい |
| **Codex** | OpenAI | Web / API | クラウドサンドボックス実行。プラグイン追加でブラウザ操作も可能 |
| **Cursor** | Cursor Inc. | 専用エディタ | ブラウザ機能が強い。URLや実画面を見ながら細かく検証しやすい |
| **GitHub Copilot** | GitHub/MS | IDE拡張 | コスパが良い。複数モデルを使え、GitHub連携が非常に自然で導入が最も楽 |

---

## これからコーディングエージェントを選ぶとしたら

| ツール | 向いていること | 補足 |
|------|--------------|------|
| **Claude Code** | 要件整理・文脈理解・運用設計まで含めて主導したい場面 | 機能面の先行投入が多く、新しい体験を最初に使いたいならここを見る価値が高い |
| **Codex** | 実装を前に進める・一つの正解に収束させる・複数案件を回す | 単純なコーディング能力と自律性が高い。ブラウザ操作をしたい場合はプラグイン拡張を推奨 |
| **Cursor** | UI実装や画面検証を細かく詰めたい場面 | ブラウザ機能を活かして、実画面を見ながら修正サイクルを回しやすい |
| **GitHub Copilot** | まず低コストで始めたい・GitHub中心の現場 | 導入負荷が低く、複数モデルも選べるので最初の一手として強い |

<div class="accent">

**おすすめの使い分け:** 先端機能はClaude Code、実装はCodex、画面検証はCursor、導入のしやすさはGitHub Copilot

</div>

---

## AIの課題①: ハルシネーション

<div class="warning">

**ハルシネーション**: AIが「もっともらしい嘘」をつくことがある

</div>

**例:**
- 存在しないライブラリ名を提案する
- 架空のAPI仕様を説明する
- 間違ったコードを「正しい」と言い張る

**防止のコツ:**
- 事実確認が必要な内容は「出典を示して」と依頼する
- 重要な判断は「根拠・前提・不確実な点を分けて答えて」と指定する
- 生成結果をそのまま使わず、公式ドキュメント・実機・人間レビューで検証する
- 一発で完成させようとせず、レビュー→修正の往復を前提にする

**大手企業への提案時には特に注意** — 誤った情報を提示すると信頼を失う

---

## AIの課題②: セキュリティ

<!-- _class: xsmall -->

<div class="warning">

**無料プランは業務利用で使わない**

入力内容を気をつけても、やり取り全体から機密情報が混ざるリスクは残ります。  
顧客案件・社内情報・認証情報を扱う場面では、**無料プランは避ける**ことを前提にします。

</div>

### プランごとの考え方

| プラン | 学習利用 | 備考 |
|--------|---------|------|
| 無料 | **利用しない** | 業務データ・顧客情報は入力しない |
| 有料（API / Team / Enterprise） | **原則されない** | 商用利用では学習利用を禁止する契約が基本 |
| 有料（API / Team / Enterprise） | 30日程度の保持はあり得る | 不正利用検知・障害調査などのために一時保持される場合がある |

### セキュリティの鉄則

- **APIキーは他人に教えない**（クレジットカードと同等）
- 大手顧客の機密データは**原文のまま入れない**
- **完全にゼロリスクではない**と理解する

---

## セキュリティのベストプラクティス

<!-- _class: small -->

### 検証環境

- ローカル端末にAPIキーを置いたまま検証しない
- Secretは **GCP Secret Manager / AWS Secrets Manager / Vercel Env** に保存
- 実行環境は **Cloud Run などのクラウド側** に寄せる

### AIに渡す権限

- **Vercel / GCP にフルアクセスできるコマンドは渡さない**
- 初回だけ人手で環境変数を登録し、以後はAIから参照できない設計にする
- IAMロールやCLI権限を絞り、**環境変数の読取権限を持たせない**

> 現状もっとも現実的なのは「シークレットはクラウド保管」「AIには最小権限のみ付与」の運用

---

## AIの課題への対策まとめ

**ベストプラクティス:**

1. 重要な情報は必ず**人間が検証**する
2. AIの出力は**別プロセスでレビュー**させる
3. **複数回のやり取り**で精度を上げる
4. **具体的な指示**を出す（曖昧な指示 → 曖昧な回答）
5. **有料プラン**を使用してセキュリティを確保
6. 顧客の機密情報は**絶対にAIに入力しない**

> AIが書いたコードの責任は人間にある

# セッション2
## プロンプトエンジニアリング（30分）

---

## プロンプトとは？

AIに対する **「指示文」** のこと

```
あなた（人間）  →  プロンプト（指示）  →  AI  →  出力
```

<div class="warning">

プロンプトの品質 = AIの出力品質。**指示が曖昧だと、出力も曖昧になる**

</div>

> プロンプトは「盛る」のが命

---

## 同じタスクでもプロンプトで結果が激変する

**❌ 曖昧な指示:**
```
チャットボット作って
```
→ 何を作ればいいかAIが判断できない。汎用的すぎるものが出てくる

**✅ 具体的な指示:**
```
大手製造業の社内ヘルプデスク用チャットボットを作成してください。
- ユーザーはログインして使用する
- チャット履歴がユーザーIDに紐づいて保存される
- 次回ログイン時に前回の続きから会話できる
- 社内FAQデータベースを参照して回答する
```
→ 明確な要件でAIが正しく動ける

---

<!-- _class: small -->

## 良いプロンプトの5つの基本要素

| 要素 | 説明 | 例 |
|------|------|-----|
| **役割** | AIに何者として振る舞うか | 「あなたはフルスタックエンジニアです」 |
| **タスク** | 何をしてほしいか | 「社内ヘルプデスクアプリを作成して」 |
| **出力形式** | どんな形で返すか | 「要件一覧をMarkdownテーブルで出力」 |
| **制約条件** | 守るべきルール | 「Next.js + Supabaseで実装」 |
| **コンテキスト** | 背景情報 | 「大手製造業の社内IT部門で使用」 |

> すべてを毎回入れる必要はないが、**タスク + 制約条件** は必須

---

## プロンプトのテンプレート

```
# 依頼
[タスク]を行ってください。

## 役割
[役割]

## 背景
[コンテキスト]

## 要件
- 要件1
- 要件2
- 要件3

## 制約条件
- 制約1
- 制約2

## 出力形式
- 形式の指定
```

**基本は必ずMarkdownで構造化**する

- 見出し、箇条書き、表で情報の意味を明確にする
- AIに貼り付ける情報を「ただの文章の塊」にしない
- **他AIの出力をそのまま渡す時だけ**、XMLタグで境界を切る

```
## 参考メモ
<note>
# 競合調査メモ
- A社は初期費用が安い
- B社は多拠点管理に強い
</note>
```

→ **Markdownを基本形にし、混ざりやすい情報だけXMLタグで囲う**のが実務では扱いやすい

## プロンプト改善テクニック①: 構造化を徹底する

AIが読みやすい形に整理すると、出力品質が安定する

```
悪い例:
在庫管理アプリ作って。CSVも必要。工場ごとに見たい。営業が見せる提案も欲しい。

良い例:
## 依頼
在庫管理アプリの提案用モックを作成してください。

## 利用者
- 工場担当者
- 本社管理者

## 必須機能
- 在庫一覧
- 拠点別フィルタ
- CSV出力

## 出力形式
- 画面構成
- 主要ユースケース
- 提案時の訴求ポイント
```

<div class="accent">

**Step by Stepをプロンプトに埋め込みすぎるのは今はアンチパターン**

モデル側が十分に考えられるので、まずは**情報を整理して渡す**方が効きやすい

</div>

## プロンプト改善テクニック②: 出力例の提示

```
以下の形式で出力してください。

### 出力例
| 顧客名 | 業種 | 課題 | 提案内容 | 概算費用 |
|--------|------|------|---------|---------|
| A社 | 製造業 | 在庫管理 | 在庫管理アプリ | 月額5万 |
```

> **例を見せると、AIは正確に形式を再現する**

## プロンプト改善テクニック③: 外部ループでレビューする

セルフフィードバックは**1つのプロンプトの中で無理にやらせない**

一度出力を作らせ、その結果を**別プロセス**でレビューさせる方が運用しやすい

```
# 例1: Claude Codeで生成 → 再レビュー
claude -p "営業向け在庫管理アプリの提案書ドラフトをMarkdownで作成して" > draft.md
claude -p "draft.md をレビューして。漏れ・矛盾・訴求不足を指摘し、改善版もMarkdownで返して"

# 例2: Codexで生成 → 別プロセスでレビュー
codex exec "社内ヘルプデスクアプリの要件整理をMarkdownで作成" > draft.md
codex exec "draft.md をレビューし、要件漏れ・曖昧表現・不足観点を指摘して改善版を返す"
```

- 生成プロセス
- レビュープロセス
- 必要なら再生成プロセス

このように役割を分けると、**レビュー観点が明確**になる

> 1発で完璧を狙うより、生成とレビューのループを分ける方が実務で扱いやすい

## ラルフループにも触れておく

**Ralph Loop / Wiggum Loop** は、AIにタスクを1つずつ処理させ続ける運用パターン

1. PRDやタスクリストを渡す
2. AIが1タスクだけ実装・テスト・コミットする
3. プロセスを終了する
4. Bashループで再起動して次のタスクへ進む

```bash
while true; do
  claude -p "$(cat CLAUDE.md)" --allowedTools "Read,Write,Edit,Glob,Grep"
done
```

<div class="accent">

コンテキストを毎回リセットするので、長時間回しても劣化しにくい

ただし、**人間のレビューは必須**で、タスク分解とツール制限が前提

</div>

## Hooks（フック）— イベント駆動の自動処理

特定イベントに連動して**自動実行**される処理

| トリガー | アクション例 |
|---------|-------------|
| ファイル保存時 | 自動フォーマット・型チェック |
| コマンド実行前 | カスタム検証・安全確認 |
| コミット前 | テスト実行・リント |

**例: 保存時に自動フォーマット**
```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "command": "npx prettier --write $FILE_PATH"
    }]
  }
}
```

> コード品質を自動的に保つ仕組み

**例: Claude Codeの実行完了時に音を鳴らしてSlack通知**
```json
{
  "hooks": {
    "Stop": [{
      "matcher": "*",
      "command": "afplay /System/Library/Sounds/Glass.aiff && curl -X POST -H 'Content-type: application/json' --data '{\"text\":\"Claude Code の実行が完了しました\"}' \"$SLACK_WEBHOOK_URL\""
    }]
  }
}
```

- 長時間の実行終了にすぐ気づける
- 席を外していてもSlackで把握できる
- `SLACK_WEBHOOK_URL` は環境変数で管理する

> 待ち時間の見落としを減らし、非同期で作業しやすくする運用例

## SubAgent（サブエージェント）— 並列作業の委任

| `/clear` | 会話履歴をクリア |

## 付録C: トラブルシューティング

| 問題 | 対処法 |
|------|--------|
| Claude CLIが起動しない | `claude --version` でインストール確認 |
| APIキーエラー | キーの有効期限・残高を確認 |
| npm installが失敗 | Node.js v18以上か確認 |
| ローカルサーバーが起動しない | ポート3000が使用中でないか確認 |
| AIの出力がおかしい | Markdownで再構造化し、別プロセスでレビューさせる |
| デプロイが失敗 | Vercelのログ・環境変数を確認 |

**エラー報告のコツ:**
