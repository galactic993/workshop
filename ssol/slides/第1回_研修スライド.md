# セッション1
## AI・LLMの基礎講座（30分）

---

## AI = LLMの時代

現在の「AI」と呼ばれるものの多くは **LLM（大規模言語モデル）** です

- 大量のテキストデータから言語パターンを学習
- 文章生成、翻訳、要約、**コード生成**が可能
- 人間の指示（プロンプト）に基づいて動作

<div class="accent">

AIは「何でもできる魔法」ではなく、**指示の質で出力が決まる**ツール

</div>

---

## LLMの進化の歴史

| 年 | 出来事 | インパクト |
|----|--------|-----------|
| 2020 | GPT-3 登場 | 初めて「使える」レベルのテキスト生成 |
| 2022 | ChatGPT 公開 | AI革命の引き金。一般ユーザーに普及 |
| 2023 | GPT-4 / Claude 2 | コーディング能力が実用レベルに |
| 2024 | Claude 3.5 / GPT-4o | マルチモーダル（画像+テキスト）対応 |
| 2025 | Claude 4 / コーディングエージェント | **AIが自律的に開発作業を実行** |

> たった5年で「テキスト生成」から「自律開発」へ進化

---

## LLMの仕組み（簡略版）

```
入力（プロンプト）
    ↓
LLM（パターン認識 + 文脈理解）
    ↓
出力（テキスト / コード / 分析結果）
```

**ポイント:**
- 「次に来る可能性が最も高い単語」を予測し続ける
- 学習データに基づいて応答するため、最新情報は苦手
- 同じ質問でも回答が毎回異なることがある（確率的な生成）

---

## LLMが「できること」と「できないこと」

| できること | できないこと |
|-----------|-------------|
| テキスト生成・要約・翻訳 | リアルタイムの最新情報取得 |
| コード生成・デバッグ | 100%正しい回答の保証 |
| データ分析・レポート作成 | 物理的な作業の実行 |
| アイデア出し・壁打ち | 感情や意志を持った判断 |
| 設計書の作成・レビュー | **責任を取ること** |

<div class="warning">

**重要**: AIの出力は必ず人間が検証する。最終判断と責任は常に人間の仕事

</div>

---

<!-- _class: small -->

## 主要なLLM

| LLM | 開発元 | 特徴 | 得意分野 |
|-----|--------|------|----------|
| **OpenAI** | OpenAI | 最も普及・マルチモーダル | 汎用タスク・画像理解 |
| **Claude** | Anthropic | 安全性重視・長文理解 | コーディング・分析・長文処理 |
| **Gemini** | Google | Google連携・100万トークン | 検索連携・大規模データ分析 |
| **Grok** | xAI | リアルタイム性・X連携 | 最新話題の要約・対話型リサーチ |

> 各LLMに得意不得意がある。**用途に応じた使い分け**が重要

---

## AIの世代分類

| 世代 | 例 | できること |
|------|----|-----------|
| コード補完 | Copilot初期 | カーソル位置の次の行を予測 |
| チャット | ChatGPT | 質問に回答。**実行はしない** |
| **エージェント** | **Claude Code等** | **複数ファイルを横断して自律的に実装・テスト** |

<div class="accent">

我々が使うのは最新の **「エージェント」世代** — AIが自分で考えて行動する

</div>

---

<!-- _class: small -->

## 主要なコーディングエージェント

| ツール | 開発元 | 形態 | 特徴 |
|--------|--------|------|------|
| **Claude Code** | Anthropic | CLI / IDE / Web | 最も多機能。サブエージェント・MCP・Skills・Hooks |
| **Codex** | OpenAI | Web / API | クラウドサンドボックス実行。Slack連携可 |
| **Cursor** | Cursor Inc. | 専用エディタ | AI特化エディタ。複数モデル切替 |
| **GitHub Copilot** | GitHub/MS | IDE拡張 | 最も普及。GitHub連携が強力 |
| **Gemini Code Assist** | Google | IDE拡張 | GCP統合。100万トークン対応 |

---

## これからコーディングエージェントを選ぶとしたら

| ツール | 向いていること | 補足 |
|------|--------------|------|
| **Claude Code** | 要件整理・文脈理解・説明責任が必要な場面 | なぜその結論にしたかを言語化しやすく、コーディング以外もまとめて任せやすい |
| **Codex** | 実装を前に進める・一つの正解に収束させる・複数案件を回す | 単純なコーディング能力と自律性が高く、利用上限も比較的大きい |

<div class="accent">

**おすすめの使い分け:** プランニングはClaude Code、実装はCodex

</div>

---

## AIの課題①: ハルシネーション

<div class="warning">

**ハルシネーション**: AIが「もっともらしい嘘」をつくことがある

</div>

**例:**
- 存在しないライブラリ名を提案する
- 架空のAPI仕様を説明する
- 間違ったコードを「正しい」と言い張る

**大手企業への提案時には特に注意** — 誤った情報を提示すると信頼を失う

---

## AIの課題②: セキュリティ

<!-- _class: small -->

### データの再学習利用

| プラン | 学習利用 | 備考 |
|--------|---------|------|
| 無料 | **される可能性あり** | 設定でOFF可の場合も |
| 有料個人 | **設定次第** | OFFにすれば使われない |
| API / Team / Enterprise | **されない** | 商用規約で明確に禁止 |

### セキュリティの鉄則

- **APIキーは他人に教えない**（クレジットカードと同等）
- 漏洩した場合 → **キーを再生成**すればOK
- 大手顧客の機密データはAIに入力しない
- 有料/APIプランを使用する

---

## AIの課題への対策まとめ

**ベストプラクティス:**

1. 重要な情報は必ず**人間が検証**する
2. AIの出力を**AIに検証させる**（セルフフィードバック）
3. **複数回のやり取り**で精度を上げる
4. **具体的な指示**を出す（曖昧な指示 → 曖昧な回答）
5. **有料プラン**を使用してセキュリティを確保
6. 顧客の機密情報は**絶対にAIに入力しない**

> AIが書いたコードの責任は人間にある

---

# セッション2
## プロンプトエンジニアリング（30分）

---

## プロンプトとは？

AIに対する **「指示文」** のこと

```
あなた（人間）  →  プロンプト（指示）  →  AI  →  出力
```

<div class="warning">

プロンプトの品質 = AIの出力品質。**指示が曖昧だと、出力も曖昧になる**

</div>

> プロンプトは「盛る」のが命

---

## 同じタスクでもプロンプトで結果が激変する

**❌ 曖昧な指示:**
```
チャットボット作って
```
→ 何を作ればいいかAIが判断できない。汎用的すぎるものが出てくる

**✅ 具体的な指示:**
```
大手製造業の社内ヘルプデスク用チャットボットを作成してください。
- ユーザーはログインして使用する
- チャット履歴がユーザーIDに紐づいて保存される
- 次回ログイン時に前回の続きから会話できる
- 社内FAQデータベースを参照して回答する
```
→ 明確な要件でAIが正しく動ける

---

<!-- _class: small -->

## 良いプロンプトの5つの基本要素

| 要素 | 説明 | 例 |
|------|------|-----|
| **役割** | AIに何者として振る舞うか | 「あなたはフルスタックエンジニアです」 |
| **タスク** | 何をしてほしいか | 「社内ヘルプデスクアプリを作成して」 |
| **出力形式** | どんな形で返すか | 「要件一覧をMarkdownテーブルで出力」 |
| **制約条件** | 守るべきルール | 「Next.js + Supabaseで実装」 |
| **コンテキスト** | 背景情報 | 「大手製造業の社内IT部門で使用」 |

> すべてを毎回入れる必要はないが、**タスク + 制約条件** は必須

---

## プロンプトのテンプレート

```
あなたは[役割]です。

以下の要件で[タスク]を行ってください。

【背景】
[コンテキスト]

【要件】
- 要件1
- 要件2
- 要件3

【制約条件】
- 制約1
- 制約2

【出力形式】
- 形式の指定
```

→ このテンプレートに当てはめるだけで、一定品質の指示が出せる

## プロンプト改善テクニック①: ステップバイステップ

一度に全部やらせず、**段階的に指示する**

```
以下の手順で進めてください：

1. まず要件を整理して、確認事項があれば質問してください
2. 要件が固まったら、画面構成案を提示してください
3. 画面構成にOKが出たら、実装に進んでください
```

<div class="accent">

AIに「まず質問してから進めて」と指示すると、**要件の抜け漏れを防げる**

</div>

---

## プロンプト改善テクニック②: 出力例の提示

```
以下の形式で出力してください：

例：
| 顧客名 | 業種 | 課題 | 提案内容 | 概算費用 |
|--------|------|------|---------|---------|
| A社 | 製造業 | 在庫管理 | 在庫管理アプリ | 月額5万 |
```

> **例を見せると、AIは正確に形式を再現する**

---

## プロンプト改善テクニック③: セルフフィードバック

AIの出力を**AIに検証させる**ことで精度を上げる

```
先ほどの結果を再度検証してください。
以下の観点で問題がないか確認し、
修正が必要な箇所があれば修正してください：

1. 情報の正確性
2. 漏れている項目
3. 論理的な矛盾
```

- 1回目: 85%の精度
- 2回目: 95%の精度
- 3回目: 99%の精度

> **人間のダブルチェックに相当する仕組み**

## CLAUDE.md — プロジェクトの指示書

プロジェクトルートに `CLAUDE.md` を置くと、毎回自動で読み込まれる

```markdown
# プロジェクト概要
大手製造業向けの在庫管理アプリです

# 開発ルール
- 日本語で回答すること
- Next.js + Tailwind CSSを使用
- Supabaseをデータベースとして使用
- セキュリティ: 顧客データのハードコーディング禁止
- テストは必ず書くこと
```

> **毎回同じ説明をする手間が不要に。チーム全員が同じルールで開発**

---

## CLAUDE.md の活用例

<!-- _class: small -->

```markdown
# プロジェクト概要
社内ヘルプデスク用チャットボット

# 技術スタック
- フロントエンド: Next.js + Tailwind CSS
- バックエンド: Supabase (Auth + Database)
- AI: Vercel AI SDK + Claude API
- デプロイ: Vercel

# 開発ルール
- 日本語で回答
- 全てのAPIエンドポイントにエラーハンドリングを入れる
- 環境変数は .env.local に設定（ハードコーディング禁止）
- コミットメッセージは日本語で

# ディレクトリ構成
- src/app/ - ページ
- src/components/ - コンポーネント
- src/lib/ - ユーティリティ
```

## Skills（スキル）— プロンプトのコマンド化

よく使うプロンプトを **コマンド化** したもの

```
/proposal    → 提案資料の構成案を作成
/mock        → モックアプリを作成
/review      → コードレビューを実行
/deploy      → デプロイ手順を実行
```

**メリット:**
- チーム全員が同じ品質の指示を出せる
- 毎回長いプロンプトを書かなくて済む
- 社内ノウハウをコマンドとして蓄積

---

## Skillsの作り方

`.claude/skills/` にMarkdownファイルを作成するだけ

**例**: `.claude/skills/mock.md`

```markdown
# モックアプリ作成

以下の手順でモックアプリを作成してください：

1. まず要件を確認し、不明点があれば質問する
2. 画面構成案を提示して承認を得る
3. Next.js + Tailwind CSS + Supabaseで実装
4. ローカルで動作確認
5. Vercelにデプロイ
6. 完了後、デプロイURLを報告
```

→ `/mock` で呼び出せる

## Hooks（フック）— イベント駆動の自動処理

特定イベントに連動して**自動実行**される処理

| トリガー | アクション例 |
|---------|-------------|
| ファイル保存時 | 自動フォーマット・型チェック |
| コマンド実行前 | カスタム検証・安全確認 |
| コミット前 | テスト実行・リント |

**例: 保存時に自動フォーマット**
```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "command": "npx prettier --write $FILE_PATH"
    }]
  }
}
```

> コード品質を自動的に保つ仕組み

## SubAgent（サブエージェント）— 並列作業の委任

メインAgentが **別のAIに作業を委任** する仕組み

```
👤 あなた → 🤖 メインAgent(Opus)
               ├→ 🤖 Agent1(Haiku): フロントエンド実装
               ├→ 🤖 Agent2(Haiku): バックエンドAPI実装
               └→ 🤖 Agent3(Haiku): テスト作成
             ↓ 結果統合 → レビュー → 完成
```

→ **並列実行で高速化**（1人でやるより3倍速い）

| モデル | 特徴 | 用途 | コスト |
|--------|------|------|--------|
| **Opus** | 最高の判断力 | 設計・レビュー・複雑なタスク | 高 |
| **Sonnet** | バランス型 | 標準的な開発作業 | 中 |
| **Haiku** | 高速・低コスト | 定型作業・調査・サブタスク | 低 |

メインAgent(Opus)が計画・指揮 → サブAgent(Haiku)が実行 → **コスト最適化**

## ハーネスエンジニアリング

### 「ハーネス」とは？

AIに仕事を任せるための**制御構造**のこと

```
人間: 要件定義 → 設計 → 指示 → レビュー → 承認
  ↕ (対話)          ↕
AI:  壁打ち  → 実装 → テスト → デプロイ
```

<div class="warning">

人間がやることは「**要件を決めること**」と「**責任をとること**」

## 従来開発 vs AI駆動開発

| 項目 | 従来開発 | AI駆動開発 |
|------|---------|-----------|
| 要件定義 | 人が書く | 人が書く + **AIと壁打ち** |
| 設計 | 人が設計 | **AIが設計案を提案** → 人が判断 |
| コーディング | 人が書く | **AIが書く** → 人がレビュー |
| テスト | 人がテスト作成 | **AIがテスト生成・実行** |
| デプロイ | 人が手動で実行 | **AIが自動でデプロイ** |

## 合格・不合格を同時に定義する

```
【合格条件（正常系）】
- ログインすると前回の会話履歴が表示される
- 質問を入力するとFAQから関連回答が返る
- 管理者はFAQの追加・編集・削除ができる

【不合格条件（異常系）】
- 未認証ユーザーがチャット画面にアクセスできてはいけない
- 他人のチャット履歴が見えてはいけない
- FAQ0件の状態でエラーにならない（「該当なし」を返す）
```

> **合格条件だけでなく「やってはいけないこと」も定義**すると、テストの精度が上がる

|------|--------|
| Claude CLIが起動しない | `claude --version` でインストール確認 |
| APIキーエラー | キーの有効期限・残高を確認 |
| npm installが失敗 | Node.js v18以上か確認 |
| ローカルサーバーが起動しない | ポート3000が使用中でないか確認 |
| AIの出力がおかしい | セルフフィードバックで検証を依頼 |
| デプロイが失敗 | Vercelのログ・環境変数を確認 |

**エラー報告のコツ:**
何をしようとしたか + エラー内容 + スクリーンショット
